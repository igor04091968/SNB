# Полный план переноса новой АТС из изоляции в продакшен (GSD-планировщик)

## 0. Цель и критерий успеха
- Цель: перевести `test-voip (10.33.1.82)` в рабочий прод-контур без деградации телефонии.
- Успех: входящие/исходящие/очереди/XMPP/переадресации/записи работают, критичных ошибок в runtime нет, есть валидный rollback.

## 1. Freeze и подготовка окна
1. Зафиксировать окно переключения (дата/время, ответственные, канал связи).
2. Ввести freeze на изменения dialplan/PJSIP/БД до конца cutover.
3. Подготовить матрицу проверок (внутренние, внешние, филиалы, XMPP, очереди, *72/#72).
4. Зафиксировать go/no-go критерии и время отката (deadline rollback).

## 2. Снимки и резервные копии (обязательно)
1. Снять backup старой АТС:
- `/etc/asterisk/*`
- дамп `snb_asterisk` (включая `sippeers`, `schedule`, `redirect`, служебные таблицы)
- текущие скрипты в `/etc/asterisk/*.sh`
2. Снять backup новой АТС в том же объёме.
3. Снять текущий AstDB с обеих сторон (`database show` + dump файла AstDB).
4. Проверить, что бэкапы читаются и доступны для быстрого восстановления.

## 3. Предпрод-паритет (до переключения трафика)
1. Сверить версии критичных файлов:
- `extensions.lua`
- `extensions_custom.lua`
- `pjsip.conf`, `queues.conf`, `xmpp.conf`, `manager.conf`
2. Проверить загрузку модулей:
- `pbx_lua.so`, `res_xmpp.so`, ODBC-модули.
3. Проверить соединения:
- БД (`snb_asterisk`)
- XMPP-клиенты (оба статуса `Connected`)
4. Проверить наличие всех endpoint/контекстов/identify для филиалов.
5. Проверить, что функции маршрутизации внешних вызовов не сводят мобильные в `PJSIP/<номер>`.

## 4. Dry-run функционала на новой АТС (до прод-трафика)
1. Внутренние звонки:
- 3/4-значные номера, занято/не отвечает/недоступен.
2. Внешние:
- местные/межгород, филиалы через транки.
3. Очереди:
- вход в очередь, распределение, fallback.
4. XMPP/Miranda:
- `MESSAGE(body)=157`, `666`
- slash payload для `from_xmpp_dial`
5. Переадресация:
- `*72` установка, `#72` снятие
- цепочка `A->B->C` (без петли)
- попытка петли (`A->B`, `B->A`) должна блокироваться.
6. Конференции:
- статические и динамические.
7. Записи:
- one-touch, авто-запись, путь сохранения, права доступа.

## 5. Техническое переключение (cutover)
1. В начале окна: включить расширенный runtime-лог (`verbose/debug`) на новой АТС.
2. Переключить маршрутизацию трафика на новую АТС:
- SIP trunks / NAT / firewall / маршруты / DNS-SRV (по фактической схеме).
3. Проверить регистрацию всех критичных endpoint после переключения.
4. Выполнить smoke-test в первые 10 минут:
- входящий внешний -> внутренний
- внутренний -> внешний
- XMPP-клик -> вызов
- вызов через очередь.

## 6. Go/No-Go контроль (первые 30–60 минут)
1. Go, если:
- нет массовых `FAILED/CONGESTION/CHANUNAVAIL`
- XMPP сообщения доходят и originate исполняется
- очереди и внешние вызовы стабильны
- нет критичных SQL/ODBC ошибок в runtime.
2. No-Go (немедленный rollback), если:
- критичный сценарий недоступен более N минут
- массовая деградация > оговорённого порога
- повреждение CDR/CEL или потеря маршрутизации.

## 7. Rollback (жёсткий, заранее подготовленный)
1. Остановить приём нового трафика на новой АТС.
2. Вернуть SIP/DNS/маршруты на старую АТС.
3. Перезапустить сервисы старой АТС и подтвердить регистрации.
4. Прогнать минимальный smoke-test старой АТС.
5. Зафиксировать причину отката и заморозить повторный cutover до RCA.

## 8. Пост-cutover стабилизация (T+24/T+72)
1. Мониторить:
- runtime логи, CDR/CEL, статус XMPP, ODBC, очереди.
2. Проверить бизнес-сценарии по расширенной матрице.
3. Снять baseline:
- active calls, ASR/ACD, проценты отказов, топ-ошибки.
4. Выключить повышенный debug после стабилизации.

## 9. Обязательные артефакты после завершения
1. Протокол cutover (время, шаги, результаты, отклонения).
2. Таблица PASS/FAIL по acceptance-кейсам.
3. Зафиксированный rollback-пакет (актуальный).
4. Обновлённые runbook/инструкции для дежурных.

## 10. Чек-лист исполнения (короткий)
1. Freeze + окно + роли.
2. Полные backup старой/новой АТС.
3. Предпрод-паритет конфигов и модулей.
4. Dry-run всех критичных сценариев.
5. Cutover трафика.
6. Go/No-Go в первые 60 минут.
7. При риске — rollback без дискуссий.
8. Пост-стабилизация и отчёт.
